# AI-World MVP 产品需求文档 (PRD)

## 1. MVP 核心定位与原则

**产品核心**：一个极简、沉浸式的“AI 引导思考画布”，帮助用户把一次高难度沟通从“凭感觉”变成“可推演、可落地”的策略与话术。

**北极星指标**：单会话高质量卡片交付率（用户在一次会话内产出可复制/可导出的执行卡片；“高质量”可用 Rubric 总分阈值判定，如 ≥70）。

**MVP 成功标准（建议口径）**：
- 在较短时间（如 5-10 分钟）内产出可直接使用的沟通方案（可复制/可导出）。
- AI 批注采纳率（用于验证“被引导”而非“被打断”）。

**舍弃清单 (Not to do in MVP)**：账户成长体系、数字档案认证（模块三）、企业端生态、通用技能培训库。

**目标用户（MVP）**：
- 需要在“老板/HR/跨部门/客户”场景中完成关键沟通，但缺乏策略与措辞框架的职场知识工作者。
- 典型心智：怕冲突升级、怕说错、怕被拿捏，往往会写出“自嗨/解释过度/攻击性过强或过弱”的文本。

**设计原则**：
- 视觉极简：类似 Notion 或 Linear 的克制美学，大面积留白，没有花哨的按钮，让用户专注“思考”。
- 无感 AI：AI 不作为“聊天机器人”存在，而是作为画布上的“幽灵协作者”，只在用户卡住或思考不周时，以“批注/提示卡/角色模拟”的形式出现。
- 可控与可撤销：任何 AI 自动补全、改写、推断都必须可追溯（标记来源）并可一键撤销。

## 2. MVP 选定场景与底层模型 (Schema & Rubric)

场景：【搞定一次高难度的职场沟通/谈判】

为什么选这个：沟通最怕“自嗨”，AI 能提供最有价值的“第三方视角”和“对手视角”。

任务定义：一次任务 = 目标对象 + 我的诉求 + 可接受底线 (BATNA) + 预判阻力 + 可执行话术/策略输出。

A. 任务底层结构 (Task Schema)

AI 不会直接让用户写稿，而是引导用户填充画布上的几个核心模块：

【对手画像】：他是谁？他的核心 KPI 是什么？他现在的压力在哪里？

【我的底牌】：我的核心诉求是什么？我的最差接受结果 (BATNA) 是什么？

【预判阻力】：他最可能拒绝的 3 个理由是什么？

【核心策略】：我们的共同利益点在哪里？

【交付成果】：破冰开场白、核心话术矩阵、应急应答库。

可选输入（用于提升准确度，但不作为 MVP 强依赖）：
- 对方的原话/邮件/聊天记录（粘贴文本）。
- 你已经写过的草稿（让 AI 以“批注”方式优化，而不是从零代写）。

B. 成功衡量标准 (Success Rubric)

AI 对生成内容的评价维度（也是微调大模型的 Prompt 基础）：

换位同理度 (25%)：话术中是否包含了对方利益的考量（从“我要”变成“我们能”）。

逻辑清晰度 (25%)：诉求是否明确，是否有具体的数据/事实支撑。

语气适宜度 (25%)：态度是否不卑不亢，既坚定又不引发敌意。

冲突防御力 (25%)：对潜在拒绝理由是否做出了有效的提前防御。

Rubric 呈现方式（MVP）：
- 本次任务总分（0-100）+ 四维雷达/进度条。
- 每一维给 1 句“改进抓手”（与批注联动）。

## 3. 核心功能与用户体验 (UX) 设计

我们将传统的“分栏聊天室”抛弃，采用**【无限协作画布 (Infinite Canvas)】** + **边注式 AI**的形式。

核心流程（Happy Path）：
1. 用户从极简入口选择场景或输入目标。
2. 系统创建任务画布，并给出第一张引导卡片（从轻问题开始）。
3. 用户填写卡片；当信息过浅/矛盾/缺失时，AI 以“幽灵批注”发出追问。
4. 卡片按 Schema 渐进展开，直到关键模块齐全。
5. 生成沟通方案草稿，并展示 Rubric 评分；AI 以批注推动二次打磨。
6. 进入“对手视角彩排”（可选），最后导出/复制执行版。

模块一：沉浸式极简入口 (Zen Onboarding)

页面呈现：一个类似 Google 搜索的极简页面。只有一句欢迎语和一个输入框。

文案：“今天，你想搞定什么难题？”

预设场景卡片（Tag）：[向老板提加薪]、[拒绝平级甩锅]、[催促跨部门进度]、[争取新项目资源]。

交互：用户点击 Tag 或输入文字后，画面平滑过渡，展开为“工作画布”。

AI 出现规则（“无感 AI”的可执行约束）：
- 默认不弹窗打断；优先用“边缘气泡/批注”提示。
- 触发条件：回答过短、关键字段缺失、用户停顿、或用户主动点击“我卡住了”。
- 节流：同一张卡片的追问在短时间内不重复；用户可临时关闭提示（本次任务有效）。

模块二：AI 引导式协作画布 (The Canvas) —— MVP 的绝对核心

这是用户 90% 的停留区域。它不是一个文档，而是一个结构化的动态白板。

功能节点与交互逻辑：

卡片式渐进引导 (Progressive Disclosure)：

画布上不会一开始就出现所有问题，避免用户产生心理负担。

第一幕：画布中央只有一个卡片：“告诉我，你这次沟通的目标对象是谁？他是个怎样的人？”

AI 辅助填写：如果用户写得太简单（比如“我的老板，很凶”），卡片旁边会浮现 AI 引导泡泡：“为了帮你拿下他，我需要知道：他最近在为哪些 KPI 焦虑？”

小创意（但仍可控）：语气光谱与对手强度
- 用户可选择本次沟通的“语气档位”：柔和/坚定/强硬（影响生成稿与批注建议）。
- 用户可选择“对手强度”：配合型/中立型/强硬型（影响 Red Teaming 的反问力度）。

“上帝视角”角色扮演模拟 (Red Teaming)：

当 Schema 中的【预判阻力】模块填完后，AI 会主动变身“对方角色”。

交互：画布侧边滑出 AI 模拟的“老板反馈”：“如果我听到你这么说，我会觉得你只是在抱怨工作多。你会怎么反驳我？”

这种互动让用户在真实场景中“练兵”。

Rubric 驱动的草稿生成与批注式优化 (AI-Margin Notes)：

生成：在画布右侧或下方，自动拼装生成【沟通方案/话术本】。

交互：AI 的建议不是直接重写，而是以类似 Google Docs 批注的形式出现。例如，高亮了用户的一句草稿，批注写道：“这里的语气带有防御性，建议修改为……这样能提高【语气适宜度】。”用户点击【接受】，内容自动更新。

历史与回退（MVP 版）：
- 提供撤销/重做。
- 关键节点自动生成快照（首次生成稿、每次批注批量采纳后、Red Teaming 结束后）。

一键生成行动清单 (Actionable Output)：

最终产出不仅仅是几段话，而是支持导出的 PDF/图片卡片。包括：【开场白】、【3个核心论点】、【如果对方说No的应对预案】。

轻量复盘（不进入“成长体系”）：
- 生成一张“本次关键策略卡”：你抓住了对方的哪个利益点？你设置了什么底线？你的三条核心论点是什么？
- 复盘卡可一键复制/导出，帮助用户下次复用思路。

## 4. 关键技术实现路径 (MVP 技术侧重)

前端框架：推荐使用 React-Flow 或类似白板技术的轻量级改造，支持卡片的拖拽、连线和高亮。

前端关键能力：
- 卡片节点组件：问题、输入区、状态（未填/已填/需补充）。
- 批注层：把 AI 的“建议”锚定到具体句子/字段，而不是漂浮在聊天流里。
- 导出：把“执行版话术/策略卡”导出为 PDF/图片（先做图片也可）。
- 存储：任务/画布/草稿/快照本地持久化（localStorage/IndexedDB），确保刷新不丢；后续再做登录与云同步。

AI 工程：

不需要自研大模型，接入 GPT-4o 或 Claude 3.5 Sonnet（后者在文本推理和情感拿捏上极好）。

核心工作量在于构建 Workflow Prompt：将 Schema 和 Rubric 写入 System Prompt，强制 AI 按照特定的 JSON 格式返回“引导问题”、“打分”和“批注内容”，以便前端渲染。

AI 输出 JSON 协议草案（MVP）：
- next_questions：下一步追问（用于生成新的卡片或补充提示）。
- canvas_updates：对现有卡片字段的结构化补全（需要标记来源：user/ai_infer）。
- draft：沟通方案执行稿（可含 Markdown 但必须字段化）。
- rubric：四维评分 + 各维解释。
- margin_notes：批注列表（锚点 + 建议 + 对应 Rubric 维度）。

JSON 示例（仅示意字段形态，具体可迭代）：
```json
{
  "next_questions": [
    {
      "id": "q_opponent_kpi",
      "target": "opponent_profile",
      "question": "他最近最在意的 KPI 是什么？为什么？",
      "type": "short_text"
    }
  ],
  "canvas_updates": [
    {
      "op": "set",
      "path": "opponent_profile.kpi",
      "value": "本季度营收增长",
      "source": "user"
    }
  ],
  "draft": {
    "opening": "…",
    "key_points": ["…", "…", "…"],
    "objections": [
      {
        "objection": "…",
        "response": "…"
      }
    ]
  },
  "rubric": {
    "total": 78,
    "dimensions": {
      "empathy": {"score": 20, "reason": "…"},
      "logic": {"score": 18, "reason": "…"},
      "tone": {"score": 22, "reason": "…"},
      "defense": {"score": 18, "reason": "…"}
    }
  },
  "margin_notes": [
    {
      "anchor": {"path": "draft.opening", "range": [12, 28]},
      "dimension": "tone",
      "comment": "这里语气略带防御性，建议修改为……这样能提高【语气适宜度】。",
      "suggestion": "…"
    }
  ]
}
```

成本与安全（必须写进工程约束）：
- Token 预算：每一步追问尽量短，优先结构化信息收集，再一次性生成大稿。
- 幻觉控制：对“事实/数据”标记为用户提供/未知；未知则追问，不擅自编造。
- 隐私：默认不要求上传敏感信息；用户粘贴内容时提示脱敏。

## 5. 埋点指标与验收标准 (Analytics & Acceptance)

北极星指标（MVP，建议按 session 计算）：
- 单会话高质量卡片交付率 = 达成「复制/导出」且 Rubric 总分 ≥ 阈值（如 70）的会话数 / 有效会话数
  - 会话（session）：从 `entry_viewed` 开始；30 分钟无操作或关闭页面视为结束
  - 有效会话：至少触发一次 `task_created`
  - 交付：同一会话内触发 `copy_clicked` 或 `export_clicked`
  - 高质量：以 `draft_generated` 上报的 `rubric_total` 判定；阈值可配置

关键漏斗（最少埋点）：
进入 -> 创建任务 -> 完成追问 -> 确认产出 -> 复制/导出 -> 二次复用/分享

漏斗必需事件（建议命名与口径）：
- `entry_viewed`：进入/落地页曝光
- `task_created`：创建任务（选择场景/输入目标后生成 task_id）
- `schema_completed`：完成追问/关键字段齐全（也可复用 `task_completed`，但需统一口径）
- `draft_generated`：生成执行稿（同时上报 `rubric_total`）
- `draft_confirmed`：用户确认执行稿（如 MVP 无显式按钮，可用首次 copy/export 视作确认）
- `copy_clicked` / `export_clicked`：交付动作
- `task_reused` / `share_clicked`：二次复用/分享（MVP 可先预留，后续启用）

辅助诊断事件（建议保留，用于解释漏斗掉点）：
- `task_started`
- `card_answered` / `ai_hint_shown` / `ai_hint_dismissed` / `ai_hint_accepted`
- `margin_note_accepted` / `redteam_started` / `redteam_completed`

通用属性（所有事件最小集，便于去重/归因）：
- `user_id`（未登录可用 `anonymous_id`）
- `session_id`（客户端生成）
- `task_id`
- `scenario_id`（场景模板）
- `created_at`（客户端时间）

验收标准（可开发可验收）：
- 入口页可创建任务并进入画布；刷新后任务内容不丢失。
- 画布按 Schema 渐进出现卡片；回答过短会触发 AI 提示（不弹窗强打断）。
- 可生成执行稿并展示 Rubric；批注可被逐条采纳且可撤销。
- Red Teaming 可开启/关闭；开启后 AI 以“对手角色”输出反问并引导用户完善应答。
- 导出/复制可用（至少一种可交付形式）。
- 北极星指标与关键漏斗可在分析看板中跑通（测试数据可验证）。

## 6. 风险与降级策略 (Risks & Fallback)

- 过度追问导致流失：提供“快速生成草稿”按钮（先给 60 分稿），再用批注提质。
- 模型不可用/超时：显示可重试，并允许用户继续编辑画布（离线编辑）；恢复后再生成。
- 生成不可信（编造事实）：把事实类内容标记为“需用户确认”，并在 Rubric 中扣分提示.
- 白板交互学习成本：MVP 默认不强调拖拽连线，先把“卡片渐进 + 批注”做成核心。

## 7. 开放问题 (Open Questions)

- 画布节点的最小交互集：拖拽是否必需？还是先做纵向卡片流即可？
- Rubric 权重是否允许用户选择“更看重语气/更看重逻辑”？
- 导出形态优先级：图片卡片 vs PDF vs 可复制富文本，哪种更贴近真实使用？
